环境变量¶

Node-config使用少量环境变量进行自己的配置.它还可以使用 自定义环境变量 覆盖应用程序的配置.在这两种情况下, 这些通常在加载应用程序之前在shell中导出, 但也可以在命令行或应用程序引导程序中提供.

在加载应用程序之前导出到O/S的示例:

cd /my/application
export NODE_ENV=stage
node myapp.js

传递命令行的示例:

cd /my/application
NODE_ENV=stage node myapp.js

第一次加载node-config之前在JavaScript中的示例设置:

process.env.NODE_ENV = "stage";
var config = require('config');

NODE_ENV¶

此变量包含应用程序部署环境的名称,在确定config 文件加载顺序 时表示 {deployment}. 常见的值包括 dev, stage, production-east 等,以适应您的部署策略. 可以使用 NODE_CONFIG_ENV 环境变量覆盖 NODE_ENV 的值.

NODE_ENV 的当前值在 config.util.getEnv() 中可用:

var config = require('config');
console.log('NODE_ENV: ' + config.util.getEnv('NODE_ENV'));

NODE_CONFIG_ENV¶

在某些情况下, NODE_ENV 可能不是配置识别的首选值.引入了 NODE_CONFIG_ENV 环境变量(v1.28.0), 它透明地覆盖了 NODE_ENV 的值.

NODE_CONFIG_ENV 的当前值在 config.util.getEnv() 中可用.:

process.env.NODE_CONFIG_ENV = "qa";
process.env.NODE_ENV = "production";
var config = require('config');
console.log('NODE_CONFIG_ENV: ' + config.util.getEnv('NODE_CONFIG_ENV'));

qa 配置将用于代替 production.

NODE_CONFIG_DIR¶

它包含包含您的 配置文件 的目录的路径. 它可以是从根开始的直接路径,也可以是相对路径,如果它以 ./ 或 ../ 开头.默认的 NODE_CONFIG_DIR 是 当前工作目录 下的 /config 目录,通常是你的应用程序root - 由 process.cwd() + '/config' 返回的值.

NODE_CONFIG_DIR 的当前值在 config.util.getEnv() 中可用.:

var config = require('config');
console.log('NODE_CONFIG_DIR: ' + config.util.getEnv('NODE_CONFIG_DIR'));

NODE_CONFIG¶

这允许您从命令行或shell环境覆盖任何配置. NODE_CONFIG 环境变量必须是JSON格式的字符串.此中包含的任何配置都将覆盖从配置文件中找到并合并的配置.

如果从O/S环境 和 命令行都提供了 NODE_CONFIG, 它们都将用于优于O/S环境值的命令行值.

例:

$ export NODE_CONFIG='{"Customer":{"dbConfig":{"host":"customerdb.prod"}}}'
$ node myapp.js

从“O/S环境和命令行”混合的 NODE_CONFIG 的当前值在 config.util.getEnv() 中可用:

var config = require('config');
console.log('NODE_CONFIG: ' + config.util.getEnv('NODE_CONFIG'));

HOSTNAME (or HOST)¶

此变量包含主机服务器的名称,表示 文件加载顺序.

有时, os.hostname() 返回的默认主机名不是你所期望的, 在运行你的应用程序之前设置它确保你有正确的名字.

为了向后兼容, 如果设置了 HOST 变量, 则使用该值.

无论使用什么变量或默认值, HOSTNAME 的当前值都可以在 config.util.getEnv() 中找到.:

var config = require('config');
console.log('HOSTNAME: ' + config.util.getEnv('HOSTNAME'));

NODE_APP_INSTANCE¶

如 多节点实例 部分所述,如果您正在运行应用程序的多个实例并且需要为每个实例配置不同的配置,则可以加载特定于实例的配置文件.

NODE_APP_INSTANCE 的当前值在 config.util.getEnv() 中可用.

var config = require('config');
console.log('NODE_APP_INSTANCE: ' + config.util.getEnv('NODE_APP_INSTANCE'));

ALLOW_CONFIG_MUTATIONS¶

如果此环境变量包含任何内容, 则允许配置对象的突变.这适用于测试等边缘情况, 在同一执行中改变不同场景的配置非常重要.

使用此环境变量时, 使配置对象不可变所提供的安全性会丢失.

ALLOW_CONFIG_MUTATIONS 的当前值在 config.util.getEnv() 中可用.

var config = require('config');
console.log('ALLOW_CONFIG_MUTATIONS: ' + config.util.getEnv('ALLOW_CONFIG_MUTATIONS'));

NODE_CONFIG_STRICT_MODE¶

启用严格模式时, 必须满足以下条件, 否则将在请求时抛出异常:

  • 如果设置了 NODE_ENV, 必须有一个与 NODE_ENV 匹配的显式配置文件.

  • 如果设置了 NODE_APP_INSTANCE, 必须有一个与 NODE_APP_INSTANCE 匹配的显式配置文件

  • NODE_ENV 必须与 ‘default’ 或 ‘local’ 不匹配才能避免歧义.

严格模式默认为关闭.

SUPPRESS_NO_CONFIG_WARNING¶

如果此环境变量包含任何内容, 则显示以下消息

WARNING: No configurations found in configuration directory

如果在加载node-config时未找到任何配置, 则将被禁止.这对于使用node-config的NPM模块来说非常重要, 它允许使用这些子模块的应用程序选择使用node-config(如果他们选择), 或使用不同的配置系统而不会收到恼人的警告消息.

有关在NPM模块中使用node-config的更多信息,请参阅以下文章:doc:Sub-Module-Configuration.

SUPPRESS_NO_CONFIG_WARNING 的当前值在 config.util.getEnv() 中可用.

var config = require('config');
console.log('SUPPRESS_NO_CONFIG_WARNING: ' + config.util.getEnv('SUPPRESS_NO_CONFIG_WARNING'));

自定义环境变量¶

(自1.1.0起新增)某些部署情况严重依赖于环境变量来配置最能从代码库中删除的秘密和设置. NODE_CONFIG 允许您使用JSON字符串来处理这个问题, 但有时更简单, 更平坦的环境变量更适合.

要启用自定义环境变量, 请创建一个配置文件, config/custom-environment-variables.json (或 .yaml 或 .js 或 coffee)映射环境变量命名到您的配置结构中.例如:

// config/custom-environment-variables.json
{
  "Customer": {
    "dbConfig": {
      "host": "PROD_SERVER"
    },
    "credit": {
      "initialDays": "CR_ID"
    },
    // Environment variables containing multiple configs
    // New as of config@1.14.0
    "settings": {
      "adminAccounts": {
        "__name": "ADMIN_ACCS",
        "__format": "json"
      }
    }
  }
}

…会导致 node-config 检查环境变量 PROD_SERVER 和 CR_ID.如果它们存在, 它们将 覆盖 配置中的 Customer.dbConfig.host 和 Customer.credit.initialDays 的值.

对于 ADMIN_ACCS, 它将尝试根据 _format 中指定的格式解析找到的环境变量, 并 扩展 Customer.settings.adminAccounts 的值.空环境变量将被忽略, 其映射对您的配置没有影响.

Precedence: 自定义环境变量会覆盖所有配置文件, 包括 local.json.只有命令行选项优先于它们.
