配置文件¶

配置目录¶

Node-config为正在运行的进程(通常是应用程序根目录)读取 ./config 目录中的配置文件.这可以通过将 $NODE_CONFIG_DIR 环境变量设置为包含配置文件的目录来覆盖.在加载 Node-config 之前, 它也可以从 node 设置:

process.env["NODE_CONFIG_DIR"] = __dirname + "/configDir/";
const config = require("config");

$NODE_CONFIG_DIR 可以是来自根目录的完整路径, 或者值以 ./ 或 ../ 开头则是来自进程的相对路径.

文件加载顺序¶

config目录中的文件按以下顺序加载:

default.EXT
default-{instance}.EXT
{deployment}.EXT
{deployment}-{instance}.EXT
{short_hostname}.EXT
{short_hostname}-{instance}.EXT
{short_hostname}-{deployment}.EXT
{short_hostname}-{deployment}-{instance}.EXT
{full_hostname}.EXT
{full_hostname}-{instance}.EXT
{full_hostname}-{deployment}.EXT
{full_hostname}-{deployment}-{instance}.EXT
local.EXT
local-{instance}.EXT
local-{deployment}.EXT
local-{deployment}-{instance}.EXT
(Finally, custom environment variables can override all files)

哪里

  • EXT 可以是.yml, .yaml, .xml, .coffee, .cson, .properties, .json, .json5, .hjson, .ts或.js取决于您喜欢的格式(见下文)

  • {instance} 是 多实例部署 的可选实例名称字符串

  • {short_hostname} 是你的服务器名, 直到第一个点, 来自 $HOST 或 $HOSTNAME 环境变量或 os.hostname() (按此顺序).例如, 如果你的主机名是 www.example.com 那么它将加载 www.EXT.

  • {full_hostname} 是你的整个服务器名, 当 {short_hostname} 与其他机器发生冲突时你可以使用它.

  • {deployment} 是部署名称, 来自 $NODE_ENV (或者如果指定的话, $NODE_CONFIG_ENV)环境变量

default.EXT 文件旨在包含其他文件可能覆盖的所有配置参数.覆盖是在逐个参数的基础上完成的, 因此后续文件仅包含该覆盖的唯一参数.

{hostname} 和 {deployment} 文件允许您调整特定服务器或部署的配置.这些文件旨在与版本控制系统中的其他文件一起使用.

local 文件¶

local 文件旨在 不 在您的版本控制系统中跟踪.外部配置管理工具可以在应用程序部署之前, 在加载应用程

使用 local 文件的最佳实践是避免使用会影响所有 ‘{deployment}’ 和 ‘{instance}’ 情况的全局 local.EXT 文件.相反, 选择适用的 local-{instance}.EXT, local-{deployment}.EXT 或 local-{deployment}-{instance}.EXT 的最具体变体.使用此设计, 您可以避免由于本地覆盖而导致测试在本地传递但在推送后未通过的情况.选择一个不用于测试环境的 local 文件变体！

默认 NODE_ENV¶

如果在环境中没有设置 NODE_ENV, 则使用默认值 development.

数组通过替换合并¶

请注意, 数组始终通过替换合并.也就是说, 覆盖文件中数组的完整内容将替换父文件中的完整数组.

多实例部署¶

In order to support different configurations for each application instance running on a machine, the NODE_APP_INSTANCE 环境变量 is inspected, and used for loading instance specific files.

例如, 如果设置了 NODE_APP_INSTANCE=3 环境变量, 则在加载 default.json 之后立即加载一个名为 default-3.json 的文件. .

这适用于所有配置文件, 因此如果您的 NODE_APP_INSTANCE=3 和 NODE_ENV=stage, 则在加载 stage.yaml 之后立即使用名为 stage-3.yaml 的文件如果存在, 将加载.

环境变量¶

自定义环境变量 覆盖所有配置文件, 包括local.json.只有命令行选项优先于它们.

文件格式¶

支持以下文件格式, 可通过文件扩展名识别.

在文件加载之前删除所有 // 和 /* ... */ 样式注释, 允许您添加注释到文件格式, 例如.json本身不支持注释.

Javascript对象表示法 - .json¶

以 .json 结尾的文件以JSON格式解析.例:

{
  "Customer": {
    "dbConfig": {
      "host": "localhost",
      "port": 5984,
      "dbName": "customers"
    },
    "credit": {
      "initialLimit": 100,
      "initialDays": 1
    }
  }
}

Javascript 5对象表示法 - .json5¶

以 .json5 结尾的文件以 JSON5 format进行解析.例:

{
  Customer: {
    dbConfig: {
      host: "localhost",
      port: 5984,
      dbName: "customers",
    },
    credit: {
      initialLimit: 100,
      initialDays: 1
    }
  }
}

.json5解析器是node-config的依赖项.这是唯一的自定义文件格式, 不需要将其作为依赖项添加到应用程序.

人类JSON - .hjson¶

以 .hjson 结尾的文件在 Hjson format中解析.例:

{
  Customer:
  {
    dbConfig:
    {
      host: localhost
      port: 5984
      dbName: customers
    }
    credit:
    {
      initialLimit: 100
      initialDays: 1
    }
  }
}

.hjson解析器不是node-config的依赖项.应用程序的 package.json 文件的 dependencies 部分必须包含 hjson 模块才能读取和解析.hjson文件.

Yaml ain’t Markup Language - .yaml¶

以 .yaml 或 .yml 结尾的文件以YAML格式解析. Node-config不包含YAML库, 请注意有一些好的库和一些不那么好的库.如果没有可用的全局 Yaml 变量, node-config将尝试加载一个好的变量(js-yaml).例:

// Customer module configs
Customer:
  dbConfig:
    host: localhost
    port: 5984
    dbName: customers
  credit:
    initialLimit: 100
    // Set low for development
    initialDays: 1

应用程序的 package.json 文件的 dependencies 部分必须包含yaml解析器, 而 js-yaml 是建议的解析器.

JavaScript 模块 - .js¶

以 .js 结尾的文件将作为JavaScript模块加载并运行.模块必须导出配置对象. JavaScript模块支持具有“延迟”和“原始”值的功能.

See JavaScript配置文件的特殊功能 for details.

CoffeeScript模块 - .coffee¶

你知道你喜欢什么.以 .coffee 结尾的文件将作为CoffeeScript模块加载并运行.示例CoffeeScript配置文件:

module.exports =
  # Customer module configs
  Customer:
    dbConfig:
      host: "localhost"
      port: 5984
      dbName: "customers"
    credit:
      initialLimit: 100
      // Set low for development
      initialDays: 1

CoffeeScript不是node-config的依赖项.应用程序的 package.json 文件的 dependencies 部分必须包含coffee-script才能读取CoffeeScript配置文件.

CoffeeScript对象表示法 - .cson¶

以 .cson 结尾的文件以CoffeeScript对象表示法加载和解析, 因为你永远不会有足够的文件格式.示例CSON配置文件:

{
  #comment
  Customers: {
    dbName: 'customers'
    dbPassword: 'pw1'
    lang: [
      'en'
      'es'
    ]
  },
  AnotherModule: {
    param4: "value4"
  },
  staticArray: [
    2
    1
    3
  ]
}

CSON不是node-config的依赖项.应用程序的 package.json 文件的 dependencies 部分必须包含 cson 模块才能读取CoffeeScript对象表示法配置文件.

属性文件格式 - .properties¶

以 .properties 结尾的文件在属性文件表示法中加载和解析, 这是一种在许多非JavaScript实用程序中很流行的点式名称间距样式.示例.properties配置文件:

# Comment
Customer.dbConfig.host = localhost
Customer.dbConfig.port = 5984

每个配置最终都会成为一个字符串, 但这在javascript中并不是特别麻烦.

.properties解析器不是node-config的依赖项.应用程序的 package.json 文件的 dependencies 部分必须包含 properties 模块才能读取和解析.properties文件.

TOML文件格式 - .toml¶

以 .toml 结尾的文件在TOML(Tom’s Obvious, Minimal Language)中加载和解析, 这是一种新的非常简单的配置文件格式.示例.toml配置文件:

[database]
host = "localhost"
port = 5984

.toml解析器不是node-config的依赖项.应用程序的 package.json 文件的 dependencies 部分必须包含 toml 模块才能读取和解析.toml文件.

Typescript - .ts¶

以 .ts 结尾的文件将作为Typescript模块加载并运行.示例.ts配置文件:

export default {
  Customer: {
    dbConfig: {
      host: "localhost",
      port: 5984,
      dbName: "customers"
    },
    credit: {
      initialLimit: 100,
      initialDays: 1
    }
  }
};

有关更多示例, 请参阅 Typescript测试文件.

Typescript不是node-config的依赖项.应用程序的 package.json 文件的 dependencies 部分必须包含typescript才能读取Typescript配置文件.

可扩展标记语言 - .xml¶

以 .xml 结尾的文件以XML格式解析.例:

// Customer module configs
<config>
  <Customer>
    <dbConfig>
      <host>localhost</host>
      <port>5984</port>
      <dbName>customers</dbName>
    </dbConfig>
    <credit>
      <initialLimit>100</initialLimit>
      <initialDays>1</initialDays>
    </credit>
  </Customer>
</config>

应用程序的 package.json 文件的 dependencies 部分必须包含x2js才能读取XML配置文件.
